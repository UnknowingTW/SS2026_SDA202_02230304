## Diagram 1 — Interaction Overview Diagram ##
This diagram shows **how people interact with each other** in the assignment process — like a story flow with no software, just people and decisions.

**What happens step by step:**
1. **Professor** creates an assignment with rules and a due date
2. **Student** receives the assignment brief
3. Both work in parallel — student codes, professor waits
4. **Student** tries to submit their code
   - If it is **past the due date** → submission is **rejected**, story ends
   - If it is **on time** → submission goes through
5. **TurnItIn** checks if the student copied from someone
   - If **plagiarism found** → professor flags it, story ends
   - If **no plagiarism** → professor grades the work manually
6. **LMS** records the final grade
7. **Student** sees their grade and feedback

## Diagram 2 — Use Case Diagram ##
This diagram shows **what the software system must be able to do** for each type of user.

**The Actors (Users):**

| Actor | Role |
|---|---|
| **Student** | Submits code, views grades |
| **Professor** | Creates assignments, grades work |
| **Admin** | Manages users, handles audits |
| **TurnItIn** | External plagiarism website |
| **LMS** | University's existing grade system |

**The Use Cases (Features):**
- **Student can:** Submit code, view their grade history
- **Professor can:** Create assignments, set due dates, define grading rules, approve grades
- **System automatically:** Runs the code, checks plagiarism, syncs grades to LMS
- **Admin can:** Manage user accounts, pull audit reports

**The `<<include>>` arrows mean:**
- When a student submits code → the system **automatically** runs it AND checks plagiarism
- When plagiarism is checked → it **automatically** contacts TurnItIn

## Diagram 3 — Interaction Overview Diagram ## 
This is the **most detailed diagram**. It combines Diagram 1 and Diagram 2 — showing the full journey with the system helping at every step.

**What happens step by step:**

1. **Start** — system checks who is logging in
   - If **Professor** → system lets them create assignment, set criteria and due date
   - If **Student** → system shows them the assignment brief
2. **Student submits code** → system checks the timestamp
   - If **late** → system automatically **rejects** it, done
   - If **on time** → continues
3. **Two things happen at the same time (fork):**
   - System **runs the code** against test cases
   - System **checks for plagiarism** via TurnItIn
4. **Decision — were there any problems?**
   - If **yes** → professor reviews the flagged work, student is notified, done
   - If **no** → system **auto-grades** based on professor's criteria, then professor approves the final grade
5. **Two things happen at the same time (fork):**
   - Grade is **synced to LMS** (university records)
   - Everything is **saved in audit log** (for yearly regulatory check)
6. **Student** logs in and sees their grade, code output and feedback
